# Agentank Raid Helper — 版本历史

## v2.0.5 (2026-06-07) — 页面注入式侧边栏重构与死锁防御升级

> 🎯 目标：将弹窗控制台重构为零延迟网页注入式侧边栏，并提升自动化引擎容错与避险防作弊机制

### 🛠️ 新增功能与优化

- **注入式侧边栏 (Float Sidebar)**：彻底移除原弹窗 Popup，将控制台采用 position: fixed 原地注入游戏右侧。支持一键 `‹` / `›` 展开折叠。基于内存数据进行零延迟实时渲染，省去进程通信损耗。
- **启动防死锁机制**：将底部的生命周期 Promise 链重构为 `.catch().finally()` 结构。无论侧边栏在 DOM 渲染或获取 storage 时发生任何异常，挂机自动化引擎主循环都必定强制启动，消除死锁。
- **检测大厅容错降级**：在 `detectState()` 中将 `shell` 获取在未找到 `.raid-shell` 时自动降级为 `document.body`，防止网页外层样式类名改动导致状态机误判为 `UNKNOWN` 停滞。
- **无资源仓库避险等待**：当剩余星星与星屑均为 0 且无宝物出售时，自动引导进入仓库并**保持静止挂机等待**（打印 `无可用星星与星屑，留在仓库等待挂机宝物产出...` ），移除死循环大厅和仓库跳转，保障账号安全。

## v2.0.3 (2026-06-07) — 物理可见性与精准ID选择器热修复

> 🎯 目标：彻底修复由于常驻隐藏模态框导致主页“开始游戏”按钮被错误锁死不点击的问题

### 🛠️ Bug 修复与优化

- **物理渲染可见性校验**：在 `isModalShowing()` 中加入物理渲染尺寸检查 `offsetWidth === 0 && offsetHeight === 0`。这样即使模态框常驻于 DOM 树中且其父容器隐藏，也能被正确识别为隐藏状态，从而解除对主页“开始游戏”按钮的误判锁死。
- **精确模态框 ID 选择器**：抛弃模糊的 `.raid-modal` 类选择器，重构为直接检索页面上的真实模态框 ID（出击确认 `raidStartModal`、结算弹框 `raidSettlementModal`、技能强化 `raidChoiceModal`），从根本上避免弹框混淆。
- **主页出击守卫精简**：移除了模糊类名 fallback 导致的冲突，出击确认守卫直接锁定 `raidStartModal` 元素。

## v2.0.2 (2026-06-07) — 插件重载上下文失效防御热修复

> 🎯 目标：彻底解决由于 Chrome 插件重新加载导致老旧 content script 轮询抛出 `Extension context invalidated` 异常的问题

### 🛠️ Bug 修复与优化

- **扩展上下文校验 (`isContextValid`)**：引入统一的上下文合法性检测，在涉及 Chrome Storage (`chrome.storage.local`) 及 Extension API 调用前拦截检查。
- **自动循环终止机制**：若检测到当前扩展上下文已失效（插件被重载/更新），则立即、优雅地终止旧的 `setTimeout` 循环，阻止多余的网络/内存开销及报错。
- **异常捕获容错**：在 `log`、`syncStateToStorage` 等高频调用里增加 try-catch 块，确保在任何边缘场景下均不会抛出未捕获的运行时错误。

## v2.0.1 (2026-06-07) — 混合选择器容错修复

> 🎯 目标：解决在真实运行时部分 DOM 元素缺少 ID 导致无法点击的 bug

### 🛠️ Bug 修复与优化

- **混合选择器机制**：在出击主页和仓库等区域，真实运行的 DOM 按钮可能缺少静态 HTML 模板中的 ID。已将单 ID 选择器升级为 `ID + class + 文本内容` 的混合匹配策略，保证元素被 100% 成功捕获并点击。
  - 主页开始按钮：`#raidStartBtn` 容错 fallback 至 `.raid-home-start`
  - 仓库按钮：`#raidWarehouseBtn` 容错 fallback 至 `.raid-home-view button.raid-secondary` 或文本 "打开仓库"
  - 仓库返回按钮：容错 fallback 至 `.raid-warehouse-back` 或文本 "返回出击"
  - 开始确认框按钮：`#raidStartConfirmBtn` 容错 fallback 至文本 "开始游戏" / "进入地图"
- **模态框及状态检测增强**：
  - 增加模态框子元素过滤，完美区分 `开始确认框` 与 `结算奖励框`，防止状态判定混淆。
  - 仓库出售按钮匹配重构：不依赖父容器类名，直接全局过滤 button 文本，保证 "全部出售" / "出售" 稳定被捕获。

## v2.0.0 (2026-06-07) — 状态机引擎重构

> 🎯 目标：实现全自动循环出击，智能撤离，赢取更多星屑

### ⚡ 核心重写

- **状态机引擎**：`content.js` 完全重写为六状态自动化引擎
  - `MAIN_PAGE` → `START_CONFIRM` → `BATTLE` → `VICTORY_CHOICE` / `DEFEAT` → `WAREHOUSE` → 循环
  - 替代旧版简单的按钮文本扫描逻辑
- **真实 DOM 映射**：基于 `agentank.ai/raid` 真实页面 HTML 结构
  - 使用精确元素 ID（`raidStartBtn`, `raidRewardModal`, `raidEscapeBtn` 等）
  - 通过 `.raid-shell` 的 class（`is-run`, `is-warehouse`）判断页面阶段
  - 不再依赖模糊的 `textContent` 匹配

### 🧠 智能决策系统

- **撤离策略**（基于 remark.md 设计）：
  - 层 ≤ 3：总是继续挑战
  - 层 3 完成 + 无备用核心 → 撤离
  - 层 3 完成 + 备用核心 ≥ 1 → 继续到第 5 层
  - 层 5 完成 + 备用核心 < 2 → 撤离
  - 层 5 完成 + 备用核心 ≥ 2 → 继续到第 7 层
  - 层 ≥ 7 完成 → 总是撤离
- **强化技能选择优先级**：
  - 绝对优先：备用核心（只要出现就选）
  - 首次优先：未选过的技能各选一次
  - 重要度排序：自动护盾 > 宝物磁场 > 技能冷却 > 开局推进
- **仓库自动流程**：
  - 自动出售所有宝物（点击"全部出售"）
  - 星星为 0 时自动兑换 1 颗星星（只兑换最少量，保证能开始游戏）
  - 兑换后自动返回出击

### 🎨 Popup UI 升级

- **实时状态面板**：显示当前阶段、层数、星星、星屑余额
- **当前强化显示**：芯片样式展示已获得的强化技能及次数
- **战绩统计**：总出击数、胜利层数、撤离次数、失败次数、点击次数、运行时长
- **操作日志**：最近 20 条日志，按等级着色（info/warn/error/action/state）
- 移除旧版的"自动化配置"开关组（自动强化、备用核心等独立开关），改为全自动智能决策

### 🔧 技术改进

- **自适应轮询**：菜单 1.5s / 战斗 4s / 未知 3s，附 ±300ms 随机抖动
- **点击冷却**：1.8s 冷却防止重复点击
- **完整 Mouse Event 序列**：mouseover → mousedown → mouseup → click
- **状态同步**：通过 `chrome.storage.local` 实时同步游戏状态到 Popup

## v1.0.0 (2026-06-07) — 初始版本

### 功能

- Chrome Extension Manifest V3 基础框架
- 简单的按钮文本匹配 + 自动点击逻辑
- 通过 `findVisibleElementByText()` 查找页面按钮
- Popup 提供独立开关：自动强化、备用核心、开局推进、自动护盾、全部出售、兑换星星
- 基础统计：点击次数、运行时长
- 随机延迟点击（1-2.2s）模拟人类操作

### 已知问题

- 无状态感知，不知道当前处于哪个游戏阶段
- 无层级追踪，无法实现智能撤离
- 按钮匹配基于 `textContent` 容易误匹配嵌套元素
- 无仓库完整流程（出售 → 兑换 → 返回循环）
- 强化选择无优先级策略
