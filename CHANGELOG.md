# Agentank Raid Helper — 版本历史

## v2.1.1 (2026-06-07) — 强化选择与结算弹框物理可见性兜底热修复

> 🎯 目标：彻底修复在挑战胜利并弹出技能三选一强化选择框时，因模态框本身的 DOM 隐藏状态判定偏差导致逻辑被卡在 `BATTLE`（战斗中）的死锁问题

### 🛠️ Bug 修复与自动化增强

- **模态框物理子元素兜底判定**：
  - 引入了“以物理可见子元素来判断模态框是否显示”的增强机制：只要页面上出现了物理可见的 `.raid-choice`（三选一强化技能按钮），无论弹框 ID 是否匹配或 `isModalShowing` 判定如何，都强行将状态机流转到 `VICTORY_CHOICE` 状态并正确解析选择。
  - 同步为 `DEFEAT`（战败结算）增加了物理可见确认按钮 `raidLossConfirmBtn` 的检测兜底。
- **BATTLE 模式详尽诊断**：在 `BATTLE` 状态的循环轮询中注入了即时诊断日志，以便实时监控各模态框、可见子选项在控制台的输出状态。
- **全方位版本同步**：版本步进至 `v2.1.1`，并在 manifest、组件以及文档中同步刷新。

## v2.1.0 (2026-06-07) — 串行物理可见性获取与短路求值漏洞热修复

> 🎯 目标：彻底修复由于 `||` 运算符短路返回隐藏大厅按钮，导致状态机判定该按钮不可见而直接陷入 `UNKNOWN` 未知态空跑的漏洞

### 🛠️ Bug 修复

- **串行物理可见性获取**：重构了大厅开始游戏按钮和打开仓库按钮的获取逻辑。抛弃原 `const el = selectorA || selectorB || selectorC` 的短路写法，改为串行检测每一个选择器，只有当该选择器能抓取到元素**且在页面上物理可见** (`isVisible(el) === true`) 时才被采用。
- **防止同名隐藏元素干扰**：此重构成功避开了页面中可能存在的隐藏同名/同 class 残留 DOM 对 `document.querySelector` 造成的匹配污染，确保 100% 触发当前真正显示在主页大厅的出击按钮。
- **全方位版本同步**：版本步进至 `v2.1.0`，并在 manifest、组件以及文档中同步刷新。

## v2.0.7 (2026-06-07) — 同步挂载生命周期卡死热修复

## v2.0.6 (2026-06-07) — 注入侧边栏异步非阻塞重构

> 🎯 目标：彻底解决由于 chrome.storage.local.get 在页面初始化时异步挂起，导致生命周期 Promise 被卡在 Pending 状态而无法激活主循环的问题

### 🛠️ 性能与稳定性提升

- **同步非阻塞挂载 (`initSidebar`)**：将 `initSidebar()` 由 `async` 重构为纯同步执行函数。DOM 渲染和事件监听绑定在几毫秒内即可同步跑完，主循环在 `.finally()` 中能够瞬间被激活。
- **后台异步数据补丁**：将配置、开关、点击量和运行时长等 storage 数据改在后台以异步 Callback 形式去提取和回填，不阻塞 UI 渲染和引擎初始化流程。

## v2.0.5 (2026-06-07) — 页面注入式侧边栏重构与死锁防御升级

> 🎯 目标：将弹窗控制台重构为零延迟网页注入式侧边栏，并提升自动化引擎容错与避险防作弊机制

### 🛠️ 新增功能与优化

- **注入式侧边栏 (Float Sidebar)**：彻底移除原弹窗 Popup，将控制台采用 position: fixed 原地注入游戏右侧。支持一键 `‹` / `›` 展开折叠。基于内存数据进行零延迟实时渲染，省去进程通信损耗。
- **启动防死锁机制**：将底部的生命周期 Promise 链重构为 `.catch().finally()` 结构。无论侧边栏在 DOM 渲染或获取 storage 时发生任何异常，挂机自动化引擎主循环都必定强制启动，消除死锁。
- **检测大厅容错降级**：在 `detectState()` 中将 `shell` 获取在未找到 `.raid-shell` 时自动降级为 `document.body`，防止网页外层样式类名改动导致状态机误判为 `UNKNOWN` 停滞。
- **无资源仓库避险等待**：当剩余星星与星屑均为 0 且无宝物出售时，自动引导进入仓库并**保持静止挂机等待**（打印 `无可用星星与星屑，留在仓库等待挂机宝物产出...` ），移除死循环大厅和仓库跳转，保障账号安全。

- **主页出击守卫精简**：移除了模糊类名 fallback 导致的冲突，出击确认守卫直接锁定 `raidStartModal` 元素。

## v2.0.0 (2026-06-07) — 状态机引擎重构

> 🎯 目标：实现全自动循环出击，智能撤离，赢取更多星屑

### ⚡ 核心重写

- **状态机引擎**：`content.js` 完全重写为六状态自动化引擎
  - `MAIN_PAGE` → `START_CONFIRM` → `BATTLE` → `VICTORY_CHOICE` / `DEFEAT` → `WAREHOUSE` → 循环
  - 替代旧版简单的按钮文本扫描逻辑
- **真实 DOM 映射**：基于 `agentank.ai/raid` 真实页面 HTML 结构
  - 使用精确元素 ID（`raidStartBtn`, `raidRewardModal`, `raidEscapeBtn` 等）
  - 通过 `.raid-shell` 的 class（`is-run`, `is-warehouse`）判断页面阶段
  - 不再依赖模糊的 `textContent` 匹配

### 🧠 智能决策系统

- **撤离策略**（基于 remark.md 设计）：
  - 层 ≤ 3：总是继续挑战
  - 层 3 完成 + 无备用核心 → 撤离
  - 层 3 完成 + 备用核心 ≥ 1 → 继续到第 5 层
  - 层 5 完成 + 备用核心 < 2 → 撤离
  - 层 5 完成 + 备用核心 ≥ 2 → 继续到第 7 层
  - 层 ≥ 7 完成 → 总是撤离
- **强化技能选择优先级**：
  - 绝对优先：备用核心（只要出现就选）
  - 首次优先：未选过的技能各选一次
  - 重要度排序：自动护盾 > 宝物磁场 > 技能冷却 > 开局推进
- **仓库自动流程**：
  - 自动出售所有宝物（点击"全部出售"）
  - 星星为 0 时自动兑换 1 颗星星（只兑换最少量，保证能开始游戏）
  - 兑换后自动返回出击

### 🎨 Popup UI 升级

- **实时状态面板**：显示当前阶段、层数、星星、星屑余额
- **当前强化显示**：芯片样式展示已获得的强化技能及次数
- **战绩统计**：总出击数、胜利层数、撤离次数、失败次数、点击次数、运行时长
- **操作日志**：最近 20 条日志，按等级着色（info/warn/error/action/state）
- 移除旧版的"自动化配置"开关组（自动强化、备用核心等独立开关），改为全自动智能决策

### 🔧 技术改进

- **自适应轮询**：菜单 1.5s / 战斗 4s / 未知 3s，附 ±300ms 随机抖动
- **点击冷却**：1.8s 冷却防止重复点击
- **完整 Mouse Event 序列**：mouseover → mousedown → mouseup → click
- **状态同步**：通过 `chrome.storage.local` 实时同步游戏状态到 Popup

## v1.0.0 (2026-06-07) — 初始版本

### 功能

- Chrome Extension Manifest V3 基础框架
- 简单的按钮文本匹配 + 自动点击逻辑
- 通过 `findVisibleElementByText()` 查找页面按钮
- Popup 提供独立开关：自动强化、备用核心、开局推进、自动护盾、全部出售、兑换星星
- 基础统计：点击次数、运行时长
- 随机延迟点击（1-2.2s）模拟人类操作

### 已知问题

- 无状态感知，不知道当前处于哪个游戏阶段
- 无层级追踪，无法实现智能撤离
- 按钮匹配基于 `textContent` 容易误匹配嵌套元素
- 无仓库完整流程（出售 → 兑换 → 返回循环）
- 强化选择无优先级策略
